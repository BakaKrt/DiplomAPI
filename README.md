# Библиотека для клеточных автоматов

Проект написан для тех, кто не хочет заморачиваться с написанием собственных клеточных автоматов.

Исходный код/готовую библиотеку на C++ можно использовать в своих проектах на C, C++. Обёртку на C# можно использовать для работы в Unity или Godot, а так же с C#-совместимыми языками (любом другом языке, который использует ту же конвенцию соглашений вызовов cdecl)

Используются SIMD инструкции SSE 4.1 (полагаю, все процессоры x86_64 после 2010 года выпуска поддерживают)

<details>
    <summary>Немного о клеточных автоматах</summary>
    Клеточные автоматы могут использоваться в совершенно разных ситуациях: моделирование различных процессов в физике, химии, биологии, социологии, играх и т.д. Например: моделирование прохождения света через щёлку (опыт Юнга), игра "Жизнь" Джона Конвея, генерация 2D и 3D пещер, карты высот. В зависимости от правил используется окрестность фон Неймона, Мура или иные. <br>
    В библиотеке реализован поиск соседей по окрестности Мура

<details>
    <summary>Почему используются SSE инструкции?</summary>
При поиске соседей клетки идёт активное обращение к ОЗУ (порядка 70-100 нс), а так же активно используются ALU/FPU (в завимости от типа данных). В виду ограниченности количества блоков ALU/FPU (обычно 4 блока ALU и 2 FPU на одно ядро), они выступают "бутылочным горлышком", то есть процессор "простаивает" в ожидании завершения операций сложения. Можно улучшить кеш-локальность данных при сложении, складывая поблочно (в пределах одной кеш линии [обычно 64 байта], что даст хороший бонус к скорости выполнения, так как доступ к кеш линии практически бесплатен, в отличии от доступа к L1/L2/L3 кешу [в него попадут данные после чтения массива]).<br>
    Однако, при использовании SSE инструкций (SSE4.1) часть проблем уходит. Например, в одном регистре помещается до 16-и чисел типа uint8_t [то есть 1 байт], таких регистров 16, что даёт возможность складывать до 16 * 16 = 256 чисел одновременно. Решается проблема доступа к памяти, так как в регистр загружается сразу массив, а это очень быстро (не происходит точечного одиночного обращения к памяти, а лишь работа с регистрами [от 1 до 5 тактов процессора]). Используя маски, перемешивание, складывание и вычитание можно очень быстро найти сумму соседей клетки.

Результаты бенчмарка сравнивают производительность двух алгоритмов вычисления суммы значений соседних ячеек в рамках модели клеточного автомата:
1. Алгоритм 1 (наивный): реализация с использованием стандартных арифметических операций (ALU), без применения векторизации.
2. Алгоритм 2 (SIMD-оптимизированный): реализация с использованием SIMD-инструкций (SSE) и соответствующих регистров.

Параметры бенчмарка:
1. Размер массива тестовых элементов: SIZE = 100.
2. Количество итераций: iterations = 1000.
3. Тестовые данные: вектор из 100 элементов типа std::array<T, N>, где N = 112.

Тестовый стенд:
1. Процессор: Ryzen 7 5700X3D (8C/16T, 4.05ГГц, 96MB L3 общий кеш)
2. 32GB DDR4 3466МГц с таймингами 16, 20, 20, 21

Результаты выражены в количестве операций вычисления суммы восьми соседей в секунду для матрицы 7*16.
    Бенчмарк проводился с использованием библиотеки ankerl::nanobench. Время выполнения усреднено по 1000 прогонам. Результаты отражают пропускную способность каждого из алгоритмов при обработке фиксированного объёма данных.
|               ns/op |                op/s |    err% |     total | benchmark
|--------------------:|--------------------:|--------:|----------:|:----------
|        7,734,816.67 |              129.29 |    1.0% |      0.47 | `normal`
|          982,116.67 |            1,018.21 |    2.5% |      0.06 | `sse`

Чем ниже значение параметра "ns/op" (показывает время выполнения одной операции в наносекундах), тем лучше. Наивный алгоритм выполняется почти в (~7.88) раз медленнее, чем SSE!

Компиляторы имеют встроенные опции векторизации (SSE, SSE2, AVX, AVX2 и т.д.). В данном проекте включена оптимизация под SSE2 (это не значит, что компилятор точно векторизирует код, а лишь подсказывает ему, что так можно. От программиста так же требуется писать код, который компилятор сможет векторизировать)

## С#Wrapper:
везде используется using size_t = UInt64
Вся реализация завязана на **использовании памяти**, **выделенной в DLL** C++ (кроме Safe реализации), поэтому _требуется_ using или dispose() для освобождения памяти.
```c#
class HeightMap : IDisposable
```
### Конструктор:
```c#
HeightMap(size_t width, size_t height, size_t threadCount = 2, bool setRandom = true)
```
- `width` (`size_t`): Ширина создаваемой карты высот. Константа.
- `height` (`size_t`): Высота создаваемой карты высот. Константа.
- `threadCount` (`size_t`): Количество потоков, в которые будет обрабатывать метод TickMT. 0 для автоопределения по количеству потоков системы. По умолчанию 2. Нельзя изменить для уже созданного объекта!
- `setRandom` (`bool`): Стоит заполнить массив случайными значениями от 0 до 255. По умолчанию true.

### Tick в одном потоке
```c#
void Tick(size_t count = 1)
```
- `count` (`size_t`): Количество итераций. Если необходимо сделать несколько итераций за раз. По умолчанию 1.
Работает довольно быстро примерно до размера матрицы 150*150 [на процессоре Ryzen 7 5700x3d]

### Tick в многопопотоке
```c#
void TickMT(size_t count = 1)
```
- `count` (`size_t`): Количество итераций. Если необходимо сделать несколько итераций за раз. По умолчанию 1.
Работает быстро при размере матрицы более 150*150 [на процессоре Ryzen 7 5700x3d]

### Метод "Сделать хорошо"
```c#
void MakeGood(int type = 0)
```
- `type` (`size_t`): Тип. Всего 3, считая с 0 до 2


### Нормализация массива в диапазоне от 0 до 255
```c#
void Normalize()
```

## Пример использования
```c#
using C_Wrapper.Arrays;


namespace C_Wrapper
{
    using size_t = UInt64;
    internal class Program
    {
        static void Main(string[] args)
        {
            size_t SizeX = 10, SizeY = 10;
            size_t threadCount = 2;
            HeightMap map = new(SizeX, SizeY, threadCount);
            Console.WriteLine($"{map}\n");

            map.MakeGood(1);

            Console.WriteLine($"{map}\n");

            map.Dispose();

            SizeX = 1000 * 16; SizeY = 1000 * 16;
            map = new HeightMap(SizeX, SizeY, 0);
            map.TickMT(2);
            map.Normalize();
            map[0, 0] = 0; // Можно, идёт обращение напрямую к памяти
            Console.WriteLine(map[0, 0]); // Аналогично
            map.Dispose();
        }
    }
}

```
